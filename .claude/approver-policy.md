# Claude Approver — Java Policy

This file is the **source of truth** for the Approver's per-PR-type
criteria. The agent reads it on every run; changes to this file ship
through normal code review. A policy-change PR is evaluated by the
*previous* version of this policy.

Bootstrap-generated by `/development:bootstrap --claude-approver true`.
See `development/skills/bootstrap/docs/CLAUDE-APPS.md` in the plugin
family for the design and the broader Approver story.

---

## How the Approver reads this file

For every triggered PR:

1. Determine PR type per **Type detection** (below).
2. Apply **Baseline criteria** (every type).
3. Apply the **Per-type must-have criteria** for the detected type.
4. Build a risk register from the **Per-type risk factors**.
5. Calibrate confidence per **Confidence calibration**.
6. Render the verdict: `APPROVE` / `REQUEST_CHANGES` / `COMMENT`-with-reservations.

If type cannot be unambiguously determined, the Approver emits an
"ambiguous type" finding and `REQUEST_CHANGES`.

## Type detection

### Primary: conventional-commit prefix in the PR title

| Prefix (with or without scope) | Type |
|---|---|
| `feat:` / `feat(*):` | `feat` |
| `fix:` / `fix(*):` | `fix` |
| `refactor:` / `refactor(*):` | `refactor` |
| `chore(deps):` / `chore(deps:*)` | `chore_deps` |
| `chore(deps-major):` | `chore_deps_major` |
| `chore(runtime):` | `chore_runtime` |
| `security:` / `security(*):` | `security` |
| `docs:` / `docs(*):` | `docs` |
| `test:` / `test(*):` | `test` |
| `ci:` / `ci(*):` / `build:` / `build(*):` | `ci` |
| `chore:` / `chore(*):` *(excluding deps/runtime above)* | `chore` |
| `revert:` / `revert(*):` | `revert` |
| `hotfix:` / `hotfix(*):` | `hotfix` |

A `!` after the type or scope (`feat!:`, `fix(api)!:`) signals a
breaking change. The Approver treats this as a per-type flag, not a
separate type.

### Fallback: diff heuristic (when title prefix is missing)

- Diff entirely under documentation paths (`docs/`, `*.md`, `LICENSE`,
  `CONTRIBUTING.md`, `README.md`, `SETUP.md`) → `docs`.
- Diff entirely under test paths (`src/test/java/**`, `**/*Test.java`,
  `**/*Tests.java`, `**/*IT.java`) → `test`.
- Diff entirely under workflow / pre-commit / build-config paths
  (`.github/workflows/`, `.pre-commit-config.yaml`, the `plugins {}` /
  `buildscript {}` blocks of `build.gradle` / `build.gradle.kts`,
  `settings.gradle` / `settings.gradle.kts`) → `ci`.
- Only Java dependency manifests changed (`gradle/libs.versions.toml`
  version-catalog entries, the `dependencies {}` block of
  `build.gradle` / `build.gradle.kts`, `gradle.properties` dependency
  pins) → `chore_deps`. If a `Dockerfile` FROM line or the Gradle
  toolchain `languageVersion` is the only non-manifest diff, →
  `chore_runtime`.
- Otherwise: **ambiguous** — emit a finding.

### Tiebreaker: author hint (when primary and fallback disagree)

- `dependabot[bot]` → `chore_deps`, unless title declares `chore(deps-major):`.
- `github-actions[bot]` → `ci`.
- `claude-maintenance[bot]` → whichever the title prefix says; the
  maintenance pipeline always sets a correct conventional prefix.
- Anyone else: **still ambiguous** — emit a finding.

## Baseline criteria (apply to every type)

1. **CI is green at the head SHA.** Already a workflow gate, but
   re-checked at evaluation time (HEAD SHA freeze).
2. **No new findings introduced by this PR** in SonarCloud, Snyk, or
   CodeQL alerts. "Introduced" = present in the PR's head but not in
   the merge-base.
3. **PR description has `## Type`, `## Summary`, and `## Test plan`
   sections.** Missing fields are finding-level.
4. **No conflict markers in the diff.** A literal `<<<<<<< HEAD` in any
   tracked file is a critical finding (`REQUEST_CHANGES` regardless of
   type).
5. **No new `TODO` / `FIXME` without a tracking issue link.** A bare
   `TODO: fix this` is a finding; `TODO(#123): ...` is fine.
6. **No new secrets in the diff.** SonarCloud + gitleaks should already
   catch this; defence-in-depth re-check at this layer.
7. **No new dependency without compatibility-score evidence.** For
   dep-bumps, the PR body should contain the Dependabot compatibility
   score or a release-notes excerpt; absence is a finding.

## Per-type criteria

### `feat:` — new feature

**Must-haves:**

- A linked GitHub issue (`Closes #N` or `Fixes #N` in the body).
- The implementation visibly addresses the user story from the issue
  body.
- New code paths have **meaningful tests** — assertions bind to
  behaviour, not to mock return values, and not just to lines for
  coverage.
- Public API surface changes are documented in `CONTRACTS.md` (when
  the repo ships `contracts/`).

**Risk factors:**

- Public API surface increased without a `CONTRACTS.md` update.
- Tests look like coverage-farming (assertions only on Mockito mock
  return values, no behaviour assertions).
- No regression test for the specific user-story claim.
- Net added lines > 500 (large features are harder to evaluate).

### `fix:` — bug fix

**Must-haves:**

- A regression test that fails without the fix and passes with it.
- The fix addresses the root cause, not the symptom.
- Linked to the original bug report when one exists.

**Risk factors:**

- Diff narrows error handling without addressing the underlying bug
  (catch-and-swallow pattern).
- Test added doesn't actually exercise the failure mode.
- The fix touches more files than the bug's scope would suggest.

### `refactor:` — behaviour-preserving restructure

**Must-haves:**

- No public API change (semver-compatible). The Java binary/source
  API-compatibility gate (japicmp / revapi) is **a future addition**
  and is not built yet, so there is no machine-readable findings
  artifact to read. The Approver instead records an informational
  "Java API-stability gate not configured" finding and verifies
  no-public-break directly: LSP `find-references` on each touched
  public symbol plus a read of the diff (no changed method signature,
  return type, or checked-exception list on a public type). Even if a
  future gate's version-bump bypass would let it through, `refactor:`
  is non-negotiably no-break — flag a `REQUEST_CHANGES`.
- Coverage on touched files holds (drop > 0.5% is a finding).
- Diff is atomic — one refactoring intent per PR.

**Risk factors:**

- Public API changed without `refactor!:` and a matching version bump.
- Multiple unrelated refactoring intents batched together.
- Touched code's tests not re-run (CI ran but the touched module's
  tests are filtered out by a path-include rule).

### `chore(deps):` — patch / minor dependency bump

**Must-haves:**

- Dependabot-opened OR the title cites the package being bumped.
- Release-notes link present (PR body or comment).
- No `BREAKING CHANGE` markers in the release notes.
- Expected agent on a maintenance-dispatched bump:
  `java-dependabot-snyk-triage`.

**Risk factors:**

- Patch / minor with a known history of breaking changes within a
  minor (e.g. a library that ships behaviour changes under a patch
  tag).
- The version bump lives somewhere other than the declared catalog —
  the bump should land in `gradle/libs.versions.toml`,
  `gradle.properties`, or the `dependencies {}` block of
  `build.gradle(.kts)`, not scattered across both.

### `chore(deps-major):` — major dependency bump

**Must-haves:**

- Migration notes from the release notes are visibly addressed in the
  diff or PR body.
- `java-major-upgrade` ran successfully (when dispatched via
  `/development:maintenance` — confirm by checking the PR body for the
  agent's structured commit-body markers).
- All consumers of the upgraded library's public API in this repo are
  updated.

**Risk factors:**

- Migration sections in the release notes that aren't visibly handled.
- The upgrade itself introduces a breaking change on *our* exposed
  contract (per `CONTRACTS.md`).

### `chore(runtime):` — JDK or Docker base-image bump

This is a JDK toolchain or Docker base-image bump. The dedicated
`java-runtime-upgrade` agent is **a future slice ([#308])** — it is not
built yet. Until it ships, these PRs warrant human review: there is no
agent-produced structured commit body to cross-check, so the Approver
caps confidence and defers the merge call to a human.

**Must-haves:**

- The runtime version specifier is consistent across surfaces. The
  Gradle toolchain (`languageVersion = JavaLanguageVersion.of(N)`)
  matches the Dockerfile FROM line (e.g. `eclipse-temurin:N-jdk`).
- All transitive build-plugin or dependency bumps the new JDK forces
  have versions available on Maven Central / the configured repos.

**Risk factors:**

- A new JDK version that the project's build plugins (Spotless, Shadow,
  etc.) don't yet support.
- New runtime version is unreleased (early-access / pre-release tag in
  the Dockerfile FROM line or an EA toolchain).
- Confidence is capped pending the `java-runtime-upgrade` agent —
  these PRs currently warrant human review.

### `security:` — CVE / finding fix

**Must-haves:**

- A test that demonstrates the unsafe input no longer succeeds.
- Linked to the CVE, Snyk alert ID, or CodeQL alert.

**Risk factors:**

- Fix narrows the input rather than fixing the unsafe path.
- Multiple unrelated security fixes batched (each should be its own
  PR for clean audit trails).

### `docs:` — documentation only

**Must-haves:**

- Claims in the diff cross-check against current code (filenames,
  class names, method names, behaviour descriptions).

**Risk factors:**

- Code examples that wouldn't compile (the Approver can attempt to
  syntax-check them but doesn't build).
- Documentation referencing removed or renamed symbols.

### `test:` — tests only

**Must-haves:**

- Assertions bind to behaviour — not `assertTrue(true)`-style filler.
- New tests don't mock the very unit they're testing (mocking the
  class under test only verifies the mock).

**Risk factors:**

- Assertions only against Mockito mock return values.
- Test names that promise behaviour the assertions don't verify
  (e.g. `handlesEmptyList` with no empty-list case).

### `ci:` / `build:` — workflows / build config

**Must-haves:**

- No required gate weakened (coverage floor, branch-protection contexts,
  Sonar gate conditions).
- New jobs use least-privilege permissions (`permissions: contents: read`
  at the workflow level; widen only per job).

**Risk factors:**

- A required check was renamed without updating branch protection.
- New workflow runs on `pull_request_target` without `actions/checkout`
  pinning to `github.event.pull_request.head.sha` (RCE vector).

### `chore:` — cleanup / maintenance

**Must-haves:**

- Dead-code removal verified including dynamic references (reflection,
  `Class.forName`, `ServiceLoader` / `META-INF/services` entries,
  string-keyed registries).

**Risk factors:**

- Removed code is referenced by a `ServiceLoader` provider file, a
  reflective `Class.forName` lookup, or a string-keyed registry.

### `revert:` — clean revert

**Must-haves:**

- Clean revert (no follow-up commits with unrelated fixups).
- Dependents since the original commit have been considered.

**Risk factors:**

- Other PRs depend on the reverted commit.
- The original commit was a bugfix and the bug is now back.

### `hotfix:` — emergency

**Always `REQUEST_CHANGES` with "human review required".** Hotfixes are
emergencies; the Approver's confidence model is not calibrated for them
and a human must make the call. The Approver still produces a finding
list, so the human reviewer benefits from the analysis even though the
verdict is fixed.

## API stability (when `CONTRACTS.md` is present)

When the producer repo ships contract artifacts (per the polyrepo design
in `ARCHITECTURE.md` of the plugin family — issue #174), the Approver
applies these per-type additions:

- `feat:`, `fix:` → diff against `contracts/` must be non-breaking, UNLESS
  the PR title declares `feat!:` / `fix!:` AND a matching version bump
  is in the diff (`gradle.properties` `version`, the project `version`
  in `build.gradle(.kts)`, etc.).
- `refactor:` → `contracts/` MUST NOT change.
- `chore(deps-major):` → consider whether the upgrade itself forces a
  contract change downstream.

Detector tools per surface. The **Java library public-API** surface
would be checked by a binary/source API-compatibility gate built on
`japicmp` or `revapi` — but **that gate is not built yet**. Until it
ships there is no equivalent of Python's Griffe `api-stability` workflow
or its `griffe-findings.json` artifact for Java: the Approver records
the informational "Java API-stability gate not configured" finding and
verifies no-public-break from LSP `find-references` plus the diff. Other
surfaces are specified here but not yet wired — they'll be added as #174
covers each:

| Surface | Marker | Detector | Source for the Approver |
|---|---|---|---|
| REST / OpenAPI | `contracts/openapi.yaml` | `oasdiff` | TBD |
| gRPC / Protobuf | `*.proto` under `contracts/` | `buf breaking` | TBD |
| GraphQL | `*.graphql` under `contracts/` | `graphql-inspector` | TBD |
| AsyncAPI / event schemas | `contracts/asyncapi.yaml` | `asyncapi-diff` | TBD |
| **Library public API (Java)** | published JAR / `build.gradle(.kts)` | `japicmp` / `revapi` | TBD — gate not built yet |
| CLI parameters | `contracts/cli-help.txt` snapshot | line-diff | TBD |
| Env vars / config / webhooks | JSON Schema files under `contracts/` | JSON Schema diff | TBD |

A breaking diff without the matching version-bump signal is automatic
`REQUEST_CHANGES`. Because the Java gate is not built, the Approver
makes this call from the diff and LSP cross-references rather than from
a CI artifact. The bypass logic a future gate would use (`!` in title +
major version bump) is **not** the same as the Approver's per-type rules
— `refactor:` PRs still trigger `REQUEST_CHANGES` here on any public
break, because refactors are non-negotiably behaviour-preserving.

## Confidence calibration

Confidence is `HIGH` / `MEDIUM` / `LOW`. The Approver only `APPROVE`s
at `HIGH`. `MEDIUM` → `COMMENT`-with-reservations ("would approve if X
verified by a human"); `LOW` → `REQUEST_CHANGES`.

Adjustments to a default `HIGH`:

- Each unmet baseline criterion: drop one level.
- Each unmet per-type must-have: drop one level.
- Each risk factor matched: drop half a level (rounds down).
- Coverage drop on touched files > 0.5%: drop one level.
- PR has > 30 changed files OR > 1000 net added lines: cap at `MEDIUM`
  (large diffs are harder to evaluate reliably).
- Conventional-commit type cannot be determined: cap at `LOW`.
- `chore(runtime):` PRs cap at `MEDIUM` until the `java-runtime-upgrade`
  agent ([#308]) ships — JDK / base-image bumps currently warrant human
  review.

**Special case:** PRs from `claude-maintenance[bot]` with green CI and
zero new tool findings start at `HIGH` and can `APPROVE` directly — the
maintenance pipeline already ran tests + tool-specific verification, and
the bot is on the allowlist by default.

## REQUEST_CHANGES feedback loop (JSON block)

Every `REQUEST_CHANGES` verdict appends a **hidden HTML-comment JSON
block** to the review body. The `/development:maintenance` skill reads
this on the next maintenance run and dispatches the right triage agents
on the PR branch.

Schema:

```json
{
  "approver_version": "v1",
  "verdict": "REQUEST_CHANGES",
  "confidence": "LOW",
  "type_detected": "feat",
  "findings": [
    {
      "category": "test_quality | api_stability | coverage | risk | type_ambiguity | baseline | …",
      "title": "Short headline",
      "detail": "Multi-line markdown explanation",
      "suggested_agent": "java-coverage-improver | java-major-upgrade | java-semgrep-triage | java-sonar-triage | java-code-scanning-triage | null",
      "file": "path/to/File.java",
      "line": 42
    }
  ]
}
```

The block is wrapped in an HTML comment so it doesn't render in the
GitHub UI:

```markdown
<!-- claude-approver:findings
{ … the JSON above … }
-->
```

`/development:maintenance` parses the most recent Approver review's
hidden block, maps each finding's `suggested_agent` to the right triage
agent, dispatches them on the PR branch, pushes the fixes, and the
Approver re-runs on the next `check_suite: completed`.
